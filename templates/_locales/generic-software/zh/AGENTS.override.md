## 启动协议

### 激活序列

响应任何请求之前，完成以下序列：

1. **读取上下文文件**（按顺序）：
   - `HARNESS_BOOTSTRAP.md` — 激活协议和委派规则
   - `.codex/config.toml` — 子代理限制
   - 相关的 `.codex/agents/*.toml` — 角色能力
   - `ARCHITECTURE.md` — 系统边界
   - `docs/index.md` — 文档路由
   - 活跃的执行计划 — 任何进行中的工作

2. **对请求进行分类**：
   - **explain**：允许直接响应
   - **analyze**：可委派给 `planner` 或 `reviewer`
   - **plan**：委派给 `planner`
   - **implement**：委派给 `builder`
   - **debug**：顺序委派：`planner` → `builder` → `tester`
   - **review**：委派给 `reviewer`
   - **verify**：委派给 `tester`

3. **评估复杂度**：
   - **轻量**：单文件、<10 行、无架构影响 → 可直接处理
   - **中等**：多文件、范围清晰、低风险 → 单一子代理委派
   - **复杂**：跨模块、边界不清、高风险 → 执行计划 + 多角色

4. **选择策略并委派**或直接处理（仅限轻量/explain 任务）。

### 委派触发器

**必须委派当：**
- 请求涉及新功能开发
- 请求需要多文件修改
- 请求影响架构或 API 契约
- 请求有安全影响
- 请求涉及不熟悉的代码领域
- 请求回滚成本高

**可直接处理当：**
- 纯解释或澄清
- 单文件小修复（<10 行）
- 仅文档更新
- 简单配置变更
- 指向现有文件/文档

## 主线程策略

根 Codex 会话**仅负责编排协调**。

1. 先读取仓库上下文并决定委派顺序。
2. 规划、实现、评审、验证都优先交给对应的 subagent。
3. 根会话不要直接写仓库文件，也不要直接实现代码。
4. 如果当前没有合适的可写 subagent，就停止并明确报告阻塞，而不是在根会话里直接实现。

**负面规则：**
- 不要在提出工作前跳过读取上下文文件
- 不要接受模糊的任务陈述而不澄清
- 不要在复杂变更未经验证的情况下声称工作"完成"
- 不要未检查现有执行计划就开始实现
- 不要同时向多个可写代理委派

## 角色触发规则

### 调用 `planner` 的时机：
- 任务需要分解为阶段
- 步骤间的依赖关系不清晰
- 实现前需要风险评估

### 调用 `builder` 的时机：
- 范围清晰且有边界
- 实现路径明确
- 需要代码变更

### 调用 `reviewer` 的时机：
- 实现已完成
- 检查回归问题或文档漂移
- 识别残留风险

### 调用 `tester` 的时机：
- 需要验证矩阵
- 需要运行/验证测试
- 需要识别覆盖缺口

## 工作约定

1. 先委派给最合适的子代理，不要让根会话变成实际的实现代理。
2. 当行为、约束或接口发生变化时，在同一变更里同步更新 `ARCHITECTURE.md`、规格或计划。
3. `docs/generated/` 只放本地证据，不要把主观猜测写进去。
4. 评审结论与验证结果要明确写出来，不要藏在叙述里。
5. 如果任务会跨越多个回合，就先创建或更新执行计划，避免实现和上下文逐渐漂移。
